# BP미디어 기능정의서 / KMS

이 문서는 BP미디어의 공개 홈페이지, 관리자, 운영 데이터 규칙, 배포 규칙을 통합해서 관리하는 기준 문서다.
앞으로 새 기능을 개발하거나 기존 기능을 수정할 때는 이 문서를 먼저 확인하고, 구현이 끝나면 반드시 이 문서도 함께 갱신한다.

## 0. 문서 목적과 사용 순서

### 0.1 문서의 역할

#### 기능 세부 설명
- 이 문서는 “현재 사이트가 무엇을 하는가”를 정리하는 설명서가 아니라, “앞으로도 어떻게 만들어야 하는가”를 결정하는 기준서다.
- 운영자, 디자이너, 인간 개발자, AI 개발자 모두 같은 기준을 보기 위해 KMS를 단일 진실 원본(single source of truth)으로 사용한다.
- 관리자 페이지의 `기능 정의서 / KMS` 페이지가 원본이며, 저장소의 `docs/feature-definition.md`는 스냅샷이다.

#### 각주
- 구현이 문서와 다르면 코드를 맞추거나 문서를 업데이트해야 한다. “문서와 실제가 다른 상태”를 방치하지 않는다.

### 0.2 개발자가 반드시 지켜야 할 순서

#### 기능 세부 설명
- 1단계: 구현 전에 KMS 문서를 읽고 현재 규칙을 확인한다.
- 2단계: 문서에 없는 기능이라면 먼저 문서에 추가한다.
- 3단계: 구현 중간에 UI 이름, 데이터 정의, 정렬 기준, 권한 흐름이 문서와 같은지 다시 확인한다.
- 4단계: 배포 전 마지막으로 문서와 실제 화면을 대조한다.
- 5단계: 배포 후 changelog와 KMS 기준 문구가 같은지 확인한다.

#### 각주
- 이 순서를 지키지 않으면 용어 충돌, UI 명칭 불일치, 스파게티 코드, 기준 없는 예외 처리, 분석 데이터 오염이 반복된다.

### 0.3 문서 구조 규칙

#### 기능 세부 설명
- `##`는 대목차다.
- `###`는 세목차이며, 기능의 의도와 운영 목적을 설명한다.
- `####`는 실제 동작 규칙, 필드 정의, 예외 조건, 상태 변화, 개발 메모를 적는다.
- 디자인 규칙은 코드 예시와 사람이 읽는 설명을 같이 둔다.

## 1. 사이트 전체 구조

### 1.1 공개 페이지 구조

#### 기능 세부 설명
- `/` : 홈
- `/latest` : 최근 소식
- `/korea`, `/apr`, `/wosm`, `/people` : 카테고리 게시판
- `/post/:id` : 기사 상세
- `/calendar` : 일정 캘린더
- `/glossary` : 용어집
- `/glossary-raw` : 용어집 RAW 공개 보기
- `/contributors` : 도움을 주신 분들
- `/search` : 검색 결과
- `/feature/:category/:slug` : 특집 기사 컬렉션
- `/ai-guide` : AI 운영 가이드

### 1.2 관리자 구조

#### 기능 세부 설명
- `/admin.html` : 관리자 메인
- `/kms` : 관리자만 접근 가능한 기능정의서 / KMS
- 관리자 상위 그룹은 `운영 개요`, `콘텐츠`, `사이트 설정`을 기본 축으로 사용한다.
- `운영 개요` 그룹에는 `대시보드`, `분석`, `마케팅` 메뉴를 둔다.
- 기능정의서는 `사이트 설정` 세부메뉴가 아니라 독립 관리자 메뉴로 취급한다.

## 2. 공통 데이터 규칙

### 2.1 날짜 필드 정의

#### 기능 세부 설명
- `created_at` : 실제 생성 시각
- `publish_at` : 공개 기준 시각
- `updated_at` : 마지막 수정 시각
- 공개 정렬은 기본적으로 `publish_at DESC`, 없으면 `created_at DESC`
- 공개 화면 날짜 표시는 `YYYY년 M월 D일`
- 관리자 화면 날짜 표시는 `YYYY년 MM월 DD일 HH시 MM분 SS초`

#### 각주
- 방문자에게는 읽기 쉬운 일 단위 표기만 노출한다.
- 관리자에게는 감사(audit) 가능한 시각 단위 정보를 유지한다.

### 2.2 방문/조회 정의

#### 기능 세부 설명
- 운영 분석의 기준은 `site_visits` 중심으로 맞춘다.
- `누적 기사 조회수`는 마케팅/노출 기준으로 볼 수 있다.
- `방문자수`와 `조회수`는 같은 의미가 아니므로 혼용하지 않는다.
- 공유 링크에는 채널 구분을 위한 UTM을 붙인다.

### 2.3 삭제와 연관 정리

#### 기능 세부 설명
- 게시글 삭제 시 게시글 row만 삭제하면 안 된다.
- 대표 이미지, 슬라이드 이미지, 본문 연관 이미지, 조회 데이터, 공감 데이터, 기록 데이터까지 정리한다.
- 태그 삭제는 실제 사용 중인 게시글이 없을 때만 허용한다.

## 3. 디자인 시스템

### 3.1 기본 서체와 타이포

#### 기능 세부 설명
- 기본 서체는 `AliceDigitalLearning` 단일 사용을 원칙으로 한다.
- 한글 줄바꿈은 `word-break: keep-all` 우선으로 처리한다.
- 제목은 과하게 압축하지 않고, 본문은 읽기 위주의 line-height를 유지한다.

#### 코드 예시
```css
font-family: AliceDigitalLearning, sans-serif;
word-break: keep-all;
line-height: 1.7;
```

#### 각주
- 다른 폰트를 추가로 쓰는 경우는 아주 예외적인 브랜딩 실험이 아닌 이상 금지한다.

### 3.2 공통 여백 규칙

#### 기능 세부 설명
- 미세 간격: `8px`
- 기본 요소 간격: `12px`
- 카드 내부 기본 간격: `16px`
- 섹션 내부 기본 여백: `24px`
- 섹션과 섹션 사이: `24px ~ 32px`

#### 코드 예시
```css
gap: 8px;
gap: 12px;
padding: 16px;
padding: 24px;
margin-top: 24px;
```

### 3.3 버튼 위계 규칙

#### 기능 세부 설명
- Primary: 핵심 행동, 진한 배경, 흰색 텍스트
- Secondary: 보조 행동, 흰 배경 + 테두리
- Ghost: 파괴적이지 않은 작은 보조 행동
- 같은 위계의 버튼은 높이와 패딩을 통일한다.

#### 코드 예시
```html
<button class="btn-primary">저장</button>
<button class="btn-secondary">취소</button>
<button class="btn-ghost">더보기</button>
```

## 3.4 관리자 마케팅 대시보드

### 3.4.1 의도

#### 기능 세부 설명
- 운영 분석이 `방문/조회` 중심이라면, 마케팅 대시보드는 `유입 → 관심 → 기사 읽기` 여정을 읽는 용도다.
- 이 메뉴는 관리자 `운영 개요 > 마케팅`에서 연다.
- 데이터 소스는 `site_visits`이며, 공개 페이지 전체 방문을 사용한다.

### 3.4.2 화면 구성

#### 기능 세부 설명
- 상단 기간 선택: 시작일 / 종료일 / 빠른 선택
- 요약 카드: 고유 사용자, 전체 페이지뷰, Awareness, Interest, Consideration
- 고객 여정 흐름: 유입 채널 → 단계 → 대표 도착 페이지
- 스테이지 요약: 단계별 고유 사용자와 비율
- 운영 메모: 대표 유입 채널, 대표 도착 페이지, 대표 이동 경로, 재읽기 강도 높은 페이지
- 페이지 기회 맵: 고유 사용자 수, 1인당 조회 수, 페이지뷰를 조합한 버블 차트
- 대표 이동 경로: 가장 흔한 페이지 간 이동 순서

#### 각주
- 체류시간과 스크롤 깊이는 아직 수집하지 않으므로, 현재는 `유입 채널 / 단계 도달 / 대표 도착 / 재읽기 강도` 중심으로 본다.

## 4. 홈 화면

### 4.1 홈의 의도

#### 기능 세부 설명
- 뉴스 메인 페이지이면서 아카이브 관문 역할을 한다.
- 가장 먼저 보이는 정보는 `대표 기사`, `최신 소식`, `카테고리별 진입점`이다.
- 홈은 탐색, 큐레이션, 공유의 세 역할을 동시에 수행한다.

### 4.2 상단 구성

#### 기능 세부 설명
- 날짜/시간
- 언어 전환
- 검색
- 카테고리 네비게이션
- 티커

### 4.3 히어로 슬라이드

#### 기능 세부 설명
- 최대 5개 기사
- 자동 전환
- 일시정지/재생
- PC/모바일 별도 이미지 위치/확대
- 태그, 제목, 요약, CTA

#### 각주
- 히어로는 홈의 광고판이 아니라 “핵심 기사 큐레이션” 영역이다.

### 4.4 메인 스토리

#### 기능 세부 설명
- 대표 기사 1건
- 제목, 부제목, 요약, 날짜, 작성자, CTA
- PC/모바일 이미지 위치 분리

### 4.5 최신 소식 / 인기 소식 / 에디터 추천

#### 기능 세부 설명
- 최신 소식은 진입/복귀 시 다시 조회한다.
- 인기 소식은 운영 정의에 따른 인기 기사 목록이다.
- 에디터 추천은 운영자 수동 추천 영역이다.
- 카드의 버튼/태그/날짜 위치는 공통 규칙을 따른다.

## 5. 게시판 페이지

### 5.1 게시판의 의도

#### 기능 세부 설명
- 카테고리별 최신 기사 접근성 확보
- 제목 길이와 상관없이 카드 하단 메타 정렬 유지
- 공유, 공감, 날짜, 작성자 메타는 하단 고정

### 5.2 글 작성 모달

#### 기능 세부 설명
- 제목, 부제목, 본문
- 대표 이미지
- 슬라이드 전용 이미지
- 태그
- 특집 기사 묶음명
- 위치 정보
- SEO 정보

#### 각주
- 슬라이드 전용 이미지는 본문 이미지를 재활용하지 않고 별도 관리한다.

## 6. 기사 상세

### 6.1 기사 상세의 의도

#### 기능 세부 설명
- 읽기 중심 화면
- 태그와 공유를 상단에서 바로 노출
- 본문 아래 연관 정보(지도, 슬라이드, 해시태그, 특집, 유관기사)를 순차 배치

### 6.2 본문 아래 구성 순서

#### 기능 세부 설명
- 본문
- 지도
- 사진 슬라이드
- 해시태그
- 특집 기사 몰아보기
- 유관 기사

### 6.3 유관 기사 규칙

#### 기능 세부 설명
- 운영자가 최대 5개 직접 설정 가능
- 남는 수는 태그 우선, 제목 보조로 자동 추천
- 제목 오른쪽에 `YYYY-MM-DD` 표시
- 모바일은 날짜를 더 작고 연하게 표시

## 7. 특집 기사

### 7.1 특집 기사 의도

#### 기능 세부 설명
- 같은 주제의 기사 묶음을 뉴스 흐름과 별도로 묶어 보는 기능
- 카테고리별로 동작
- 기사 상세 하단에 최신순으로 노출
- 컬렉션 전용 페이지 제공

## 8. 용어집

### 8.1 용어집의 의도

#### 기능 세부 설명
- 검색 가능한 지식 사전
- `용어 / 설명 / 둘 다` 검색 대상 전환
- 검색엔진용 RAW 문서 제공

### 8.2 관리자 관점 규칙

#### 기능 세부 설명
- 긴 용어는 관리자 화면에서도 겹치지 않아야 한다.
- 한글은 단어 단위 줄바꿈, 영문은 하이픈/줄바꿈 허용
- 수정/삭제 동선은 인라인 또는 모달에서 명확히 제공한다.

## 9. 일정 캘린더

### 9.1 캘린더의 의도

#### 기능 세부 설명
- 스카우트 활동 일정을 월간/연간으로 보는 공개 안내판
- 일정 관리와 위치 정보, 관련 기사 연결을 함께 다룬다.

### 9.2 월간 보기

#### 기능 세부 설명
- 구글 캘린더처럼 주 단위 row 기준
- 여러 날 일정은 연속 bar
- 과도한 시간/날짜 표시는 제거
- 일정 클릭 시 상세 모달

### 9.3 우측 상태 패널

#### 기능 세부 설명
- 선택 달 기준 `진행중 / 개최예정 / 행사종료`
- 지역별 묶음
- KOR 기본 펼침
- 제목 클릭 시 상세 모달

### 9.4 하단 캘린더 지도

#### 기능 세부 설명
- 진행중/개최예정만 표시
- 지역색 기준 배지 사용
- 국가 단위 클러스터에 개수 숫자 표기
- 행사명 클릭 시 상세 모달

### 9.5 일정 등록/수정

#### 기능 세부 설명
- 국문 제목 우선, 원문 제목 보조
- 시작/종료일
- 시간 사용 여부
- 행사 태그
- 외부 링크
- 관련 기사 다중 연결
- 지도 검색
- 위치 이름 / 주소
- 설명

#### 각주
- 일정은 중요한 운영 기능이므로 로그인 상태와 관계없이 항상 fresh 비밀번호 확인 후 저장/수정/삭제한다.

#### 기능 세부 설명
- 관리자 페이지는 운영 밀도 높은 단일 업무 도구다.
- 클릭 가능한 정렬 기능은 `버튼만` 또는 `핸들만`이 아니라, 실제 시각적으로 보이는 인터랙션이 반드시 동작해야 한다.
- 하위 메뉴는 상위 메뉴 선택 시에만 보여야 한다.
- hover 색은 글자가 사라지지 않게 유지한다.

## 12. 개발자 체크리스트

### 12.1 구현 전 체크

#### 기능 세부 설명
- 이 기능이 KMS에 정의돼 있는가
- 문구/버튼/상태명이 기존 규칙과 충돌하지 않는가
- 날짜/정렬/권한 규칙이 이미 정의돼 있는가

### 12.2 구현 후 체크

#### 기능 세부 설명
- 문서와 화면이 같은가
- 관리자와 공개 화면이 같은 용어를 쓰는가
- 버튼 높이/폰트/간격이 공통 규칙을 따르는가
- 예외 흐름(권한 없음, 저장 실패, 삭제 차단)이 문서대로 작동하는가

### 12.3 각주

#### 각주
- 이 문서는 “설명서”가 아니라 “기준서”다.
- 앞으로 개발할 때 AI와 인간 개발자 모두 이 문서를 먼저 보고, 이 문서를 무시한 구현은 허용하지 않는다.
